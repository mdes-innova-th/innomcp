On Windows with this many files, the output is ~100 warning lines, all benign. I should have suppressed them with `git -c core.autocrlf=false add -A` for the snapshot OR set `core.autocrlf=input` for the operation. ## Lessons Learned (generalizable only)

1. **Before any `git add -A` on a foreign working tree, always run `git check-ignore -v` on the secret-shaped filenames** (`.env`, `.env.local`, `*.pem`, `id_rsa*`, `credentials*`). This is the only way to be 100% sure the ignore rules actually cover the files you have. Just because `cat .gitignore` shows the rule does not mean git will honor it for files with the wrong case on Windows or files added via `git add -f` in the past.
2. **Count untracked PATHS, not untracked FILES, when sizing a commit** — `git status --short` reports one line per untracked *path*, but `git add -A` recurses. The ratio is typically 1:2 to 1:3. Always state both: "194 untracked paths, expect 400-600 files staged."
3. **Lead with the actionable blocker, not the headline number** — when summarizing git state, the "ahead of main" count is less useful than "remote main is gone" or "DB is down." The number is a curiosity; the state is the action.
