# Accessibility & UX Audit — `ChatSidebar.tsx`

| Severity | Location | Issue | Fix |
|---|---|---|---|
| **High** | `SlideOver` overlay `<div>` | Backdrop `onClick={onClose}` has no keyboard equivalent and `aria-hidden="true"` is correct visually, but focus can fall behind the dialog when opened — focus is not trapped inside the dialog. | Use a focus-trap library (e.g. `focus-trap-react`) or implement a manual trap; restore focus to the triggering button on close. |
| **High** | `SlideOver` close `<button>` (truncated) | Likely missing accessible name (only an `×` or icon shown). If close button has no `aria-label` and only an icon/glyph, SR users hear "button" with no purpose. | Add `aria-label="Close {title} panel"` and ensure `type="button"`. |
| **High** | Sidebar root container (inferred) | No landmark role — appears to be a plain `<div>`. The primary navigation of a chat app should be a `<nav>` with `aria-label="Primary"`. | Wrap navigation in `<nav aria-label="Primary sidebar">` or `<aside aria-label="Chat history">`. |
| **High** | Chat list items (inferred from `summaries.map`) | List of chats likely rendered as `<div>`s with click handlers, no `<ul>/<li>` semantics, no `aria-current="page"` for active chat, no keyboard support. | Use `<ul role="list">` + `<li>`; each item a real `<button>` (or `role="link"`); set `aria-current="page"` on the active one; handle `Enter`/`Space`. |
| **High** | `summaries` delete action (inferred) | Delete chat button almost certainly has no confirmation for SR users and no accessible name ("🗑️" emoji or icon only). | Add `aria-label={`Delete chat ${title}`}`, wrap confirm in a `<dialog>` or `role="alertdialog"` with `aria-describedby`. |
| **High** | Sidebar collapse/toggle button (inferred from `isCollapsed`/`onToggle`) | Toggle button has no `aria-expanded`, no `aria-controls`, and no label describing the resulting state. | Add `aria-expanded={!isCollapsed}`, `aria-controls="chat-sidebar"`, and a dynamic `aria-label` ("Collapse sidebar" / "Expand sidebar"). |
| **High** | `faGear`, `faPalette`, `faQuestion`, `faSignOutAlt`, `faBriefcase`, `faUser`, `faHome`, `faKey` icons | `FontAwesomeIcon` renders an `<svg>` that is decorative when paired with text, but here they likely stand alone inside icon-only buttons. SR will read nothing meaningful. | Pass `title` prop for tooltip + add `aria-label` on the button; or add `aria-hidden="true"` on the icon and a visually-hidden text node. |
| **High** | New chat button (inferred) | Likely icon-only (`+` or `faPlus`); no accessible name. | Add `aria-label="Start new chat"`. |
| **High** | Rename chat (TODO #45, inferred) | Inline editing of `<li>` content destroys list semantics and focus. No `aria-label` on the edit input. | When editing, render an `<input aria-label="Rename chat">` with `autoFocus`, save on `Enter`, cancel on `Escape`, restore focus to the trigger. |
| **High** | `MotherStatusBar`, `RateLimitIndicator`, `UserPresence` (status) | Status conveyed only by color (e.g. `bg-emerald-500` dot for "alwaysActive") — fails WCAG 1.4.1 (use of color). SR users get no status info. | Pair color with text ("Online"/"Offline") or an `aria-label`; expose status via `role="status"` or `aria-live="polite"`. |
| **High** | `relativeTime()` strings ("just now", "2m ago") | Screen readers may read "just now" as gibberish or mispronounce. Also, time is not machine-readable, breaking WCAG 1.3.1. | Wrap in `<time dateTime={new Date(ms).toISOString()}>` so AT can compute actual time. |
| **Medium** | `SlideOver` open state | `if (!open) return null;` unmounts the dialog; transition is abrupt and focus is lost. No `aria-hidden` on the rest of the app, so SR may read both. | Render conditionally but manage `inert`/`aria-hidden` on `#__next` siblings; use a CSS transition; keep a sentinel for exit animation. |
| **Medium** | `SlideOver` `aria-label={title}` | Title may be user-supplied or contain only an icon name; `aria-label` overrides inner content. | Prefer `aria-labelledby` referencing the visible heading `<span>`, falling back to `aria-label`. |
| **Medium** | Backdrop click | `onClick` on a non-interactive `<div>` with no keyboard handler means keyboard users cannot dismiss the dialog with `Escape`. | Add a global `keydown` listener for `Escape` calling `onClose`; consider `role="presentation"` on the backdrop. |
| **Medium** | `ChatModeSelector`, `ToolsTypeSelector`, `ModelSettingsPanel`, etc. (children) | These are likely groups of buttons rendered as styled `<div>`s with no radiogroup/tablist semantics. | Use `role="radiogroup"` + `role="radio"` with `aria-checked`, or `role="tablist"` + `role="tab"`/`role="tabpanel"` with arrow-key navigation. |
| **Medium** | `NotificationCenter` trigger | Unread count badge likely rendered as a `<span>` with just a number; SR hears "3" with no context. | `aria-label={`Notifications, ${count} unread`}` and `aria-live="polite"` on the count. |
| **Medium** | Emoji-only agent/plugin/provider tiles (`🎼`, `🗺️`, `☁️`, `🇹🇭`, `🤖`, `🐙`…) | Emojis are read literally by SR ("conductor: musical score emoji"). Decorative emojis should be hidden; meaningful ones need labels. | `aria-hidden="true"` on decorative emojis; the visible name is already there, so ensure the parent has an accessible name and the emoji is hidden. |
| **Medium** | Provider env hints (`"Set OPENAI_API_KEY"`) | Rendered as plain text without association to the field it documents; SR users may not link the two. | Use `<label>` + `<input>` pattern or `aria-describedby` linking the input to the hint span. |
| **Medium** | `DashboardView`, `TaskDetailPanel`, `WorkspaceFileBrowser` mounts | Mounting heavy panels on toggle without lazy rendering and without announcing the view change hurts perceived performance and SR continuity. | `role="region"` + `aria-labelledby` on each; announce view changes with a polite live region; consider `<Suspense>` for code-splitting. |
| **Medium** | Color tokens `bg-emerald-500`, `bg-sky-400`, `bg-amber-400`, `bg-violet-400`, `bg-rose-400` on white/dark backgrounds | These mid-saturation Tailwind colors on `bg-background` may fail WCAG 1.4.11 (non-text contrast 3:1) in light mode, especially `bg-amber-400` (≈2.4:1) and `bg-sky-400` (≈2.6:1). | Audit with a contrast tool; darken to `*-600` for status dots, or add a 1px border to raise effective contrast. |
| **Medium** | `text-foreground` on `bg-background` with custom theme | Theme switching via `ThemeContext` may produce low-contrast pairings if a user-defined theme uses light fg on light bg. | Enforce a minimum contrast guard in `ThemeContext`; provide a `prefers-contrast` media query override. |
| **Medium** | Drag-to-reorder projects (likely, given `SidebarProject` with `color`) | No keyboard alternative, no `aria-grabbed`/`aria-dropeffect` on reorderable lists. | Provide `aria-label="Reorder"`, `Up`/`Down` arrow keys to move, `aria-live` announcement of new position. |
| **Low** | `BACKEND` import in this component | Unused at the component level; dead import bloats the client bundle. | Remove unused imports. |
| **Low** | `import { type ProviderMode } from "./ChatInput";` in the middle of the file | Import buried below the `interface ChatMessage` block violates module convention and can break some linters / tree-shaking. | Move all imports to the top. |
| **Low** | `font-awesome` SVG icons | `FontAwesomeIcon` ships large SVGs; without `aria-hidden="true"` and a textual label they are read as "image" by some SRs. | Set `aria-hidden="true"` on every decorative icon; rely on button `aria-label`. |
| **Low** | Truncated `className="flex h-7 w-7 items-center justify-center rounded-md t…"` | The close button's hover/focus styles are unknown; if focus ring is removed by `outline-none` without a replacement, fails 2.4.7. | Ensure `focus-visible:ring-2 focus-visible:ring-offset-2` is present. |
| **Low** | `shadow-xl` + dark mode overlay `bg-black/40` | `bg-black/40` over `bg-black` dark UI ≈ 2.8:1 contrast against sidebar — backdrop may be invisible to low-vision users. | Increase overlay opacity in dark mode (`bg-black/60`) or add a 1px border. |
| **Low** | Scrollable lists inside the sidebar | Likely no `tabindex="0"` and no `aria-label` on the scroll container; keyboard users can only scroll with mouse. | Add `tabindex="0"` + `aria-label="Chat history"` (or `role="region"`) and ensure visible scrollbars or `scrollbar-gutter`. |

## Top-3 quick wins

1. **Wrap the whole sidebar in `<nav aria-label="Primary">`** and convert every icon-only `<div onClick>` into a real `<button>` with `aria-label`. This alone removes roughly half the findings.
2. **Add a focus trap + `Escape`-to-close to `SlideOver`**, and pair every status dot with text or `role="status"` — fixes the most blocking keyboard and SR issues.
3. **Audit the five `dotColor` Tailwind classes against `bg-background`** in both themes and either darken to `*-600` or add a 1px ring; eliminates all color-contrast and "use of color" risks.
