|---|---|---|---|
| **High** | `<span className="hidden sm:inline">` | **Empty accessible name on mobile:** Tailwind's `hidden` applies `display: none`, which hides the text from screen readers. On small screens (`<640px`), the button only contains an `aria-hidden` icon and lacks an `aria-label`, resulting in an empty accessible name (screen reader trap). | Add `aria-label={label}` directly to the `<button>` element. This ensures a consistent accessible name across all screen sizes without relying on visual text. |
| **Medium** | `<div role="group">` & `<button>` | **Suboptimal ARIA pattern for mutually exclusive options:** Using `role="group"` with `aria-pressed` implies independent toggles. Since only one mode can be active at a time, a radio group pattern is semantically correct and sets proper user expectations. | Change the container to `role="radiogroup"`, change buttons to `role="radio"`, and replace `aria-pressed={active}` with `aria-checked={active}`. |
| **Medium** | `<button>` (Keyboard Nav) | **Missing arrow key navigation & roving tabindex:** Standard buttons require the `Tab` key to move between them. For a cohesive radio/toggle group, users expect `Left`/`Right` arrow keys to navigate between options, with only the active item in the tab order. | Implement a roving `tabIndex` (only `active` button gets `tabIndex={0}`, others `-1`) and add an `onKeyDown` handler to move focus via ArrowLeft/ArrowRight keys. Alternatively, use a headless UI library like Radix Toggle Group. |
| **Medium** | `<button>` (Focus styles) | **Missing explicit focus indicators:** The component relies on default browser outlines or global CSS resets. If global styles suppress outlines (common in Tailwind/shadcn setups), keyboard users will lose track of their focus position. | Add explicit focus-visible classes to the button: `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2`. |
| **Low** | `<button>` (Tooltip/UX) | **Inaccessible `title` attribute:** The `title={desc}` attribute is used to show descriptions on hover. However, `title` is not reliably accessible on touch devices (no hover state) and is often ignored or poorly announced by screen readers. | Replace the native `title` with a proper accessible Tooltip component (e.g., Radix UI Tooltip) that triggers on both hover and focus, or expose the description via `aria-describedby` if a custom tooltip is used. |
| **Low** | `cls` (Color Contrast) | **Low opacity contrast risks:** Active states use very low background/border opacities (`bg-sky-500/8`, `border-sky-500/40`). Depending on the underlying light/dark background, the 40% opacity border may fail the WCAG 3:1 contrast ratio requirement for UI components. | Increase the border and background opacities (e.g., `border-sky-500/60`, `bg-sky-500/20`) or verify contrast ratios against both light and dark mode base backgrounds using a contrast checker. |
