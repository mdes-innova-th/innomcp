## Accessibility & UX Audit: `ChartRenderer.tsx`

| Severity | Location | Issue | Fix |
|----------|----------|-------|-----|
| **Critical** | `SortableTable` – column header click handlers | Clicking a column header sorts the table, but the headers are likely rendered as `<div>` or `<th>` without `role="button"` and are not keyboard accessible. Screen reader users cannot activate the sort action. | Replace clickable elements with `<button>` inside `<th>`, or add `role="button"` and `tabindex="0"` with `onKeyDown` handling for `Enter`/`Space`. Ensure focus is moved to the header after sort. |
| **Critical** | `SortableTable` – table structure | Table may lack proper semantic elements (`<thead>`, `<tbody>`, `<th scope="col">`). Screen readers cannot associate headers with data cells. | Wrap headers in `<thead>` with `<tr>` and each header as `<th scope="col">`. Wrap rows in `<tbody>`. Add a `<caption>` if a visible title is missing. |
| **High** | `SortableTable` – missing `aria-sort` | No `aria-sort` attribute on sorted columns. Screen readers don’t announce the sort direction or current sorted column. | Add `aria-sort={sortCol === idx ? (sortDir === 'asc' ? 'ascending' : 'descending') : 'none'}` to each `<th>`. Update on sort toggle. |
| **High** | `SortableTable` – no sort announcement | When sort order changes, there is no live region update. Users relying on screen readers may not know the table has been reordered. | Add an `aria-live="polite"` region (e.g., a visually hidden `<div>`) that announces “Sorted by [column] ascending/descending” after each sort. |
| **High** | `SortableTable` – missing focus indicator | If headers are made focusable, there may be no visible focus ring (depends on browser defaults). Users navigating by keyboard cannot see which header is focused. | Ensure headers have a visible `:focus-visible` outline (e.g., Tailwind’s `focus:outline-2` or custom CSS). |
| **Medium** | `KpiCards` – grid class `sm:grid-cols-${...}` | Dynamic class string like `sm:grid-cols-${Math.min(kpis.length, 4)}` will not be scanned by Tailwind’s JIT compiler, so the class may not be generated. This can break the grid layout on small screens, causing overlapping content that is inaccessible. | Use a fixed number of columns (e.g., `sm:grid-cols-2` or `sm:grid-cols-4`) and let items wrap naturally, or apply a style object directly. |
| **Medium** | `KpiCards` – semantic grouping | KPIs are rendered as plain `<div>`s. Screen readers may not recognise them as a list of related items. | Wrap the container with `role="list"` and each card with `role="listitem"`, or use `<ul>`/`<li>` elements. |
| **Medium** | `StatsBadges` – background contrast | Badges use `bg-blue-100` with `text-blue-800` (light background/light text). | **Medium** | `SummaryCard` – non-semantic key/value display | Row and column counts are shown with `<div>`s. Screen readers may not associate the label (“Rows”) with the value. | Use `<dl>` / `<dt>` / `<dd>` or `aria-labelledby` to link label and value. |
