# Concurrency / Async-Safety Audit

The module is a pure async function with **no module-level mutable state**, so most classic concurrency classes don't apply. The real risks are I/O race conditions, path/time races, and TOCTOU issues.

| Severity | Location | Race / Issue | Fix |
|---|---|---|---|
| **High** | `analyzeData` — `fs.readFile(safePath, "utf-8")` (line ~67) | **TOCTOU race / untrusted symlink swap.** `path.resolve` + `startsWith` check is performed, but the file is *not* revalidated at `open` time. A symlink (or directory junction) created between the prefix check and `readFile` can redirect the read outside `workspaceRoot` (e.g., to `/etc/passwd` or a co-tenant file). | Use `fs.open(safePath, "r")` first, then `fstat()` and walk up `dirfd` with `fs.realpath` to confirm containment *while holding the fd*. Better: resolve relative to a `workspaceRoot` opened with `fs.opendir` and reject any path whose realpath escapes it. Reject symlinks explicitly with `S_ISLNK` if your platform exposes it. |
| **High** | `analyzeData` — artifact filename `chart-${Date.now()}.svg` (line ~108) | **Filename collision / cache-stampede on rapid calls.** Two concurrent `analyzeData` invocations within the same millisecond produce identical filenames; the second `writeFile` either silently overwrites the first's chart or interleaves bytes if the writer is shared. Combined with `fs.mkdir({ recursive: true })` (idempotent but still racy across filesystems) this is a classic stampede. | Generate a collision-free name: `crypto.randomUUID()` or `${Date.now()}-${crypto.randomUUID().slice(0,8)}.svg`. Write via `fs.writeFile(path + ".tmp", ...)` then `fs.rename()` to make the publish atomic. Optionally short-circuit duplicate `(input, opts)` requests with a `Map<key, Promise<AnalysisResult>>` in-flight cache. |
| **Medium** | `analyzeData` — `fs.mkdir(dir, { recursive: true })` + subsequent `fs.writeFile` (line ~107–110) | **Check-then-act across two awaits.** Between `mkdir` returning and `writeFile` running, another call (or external process) can `rm -rf` the directory; the `writeFile` will reject with `ENOENT`. The whole artifact section is not wrapped in any error recovery. | Wrap the mkdir+write in a retry loop (1–2 attempts) with re-`mkdir` on `ENOENT`. Or create the directory with `{ recursive: true }` *and* verify presence via `fs.stat` immediately before writing; on `ENOENT`, retry once. |
| **Medium** | `analyzeData` — `parseCSV` invoked with `text.trim()` (line ~71) | **Ordering / streaming assumption.** The function assumes the entire CSV fits in memory and is UTF-8 text returned by `fs.readFile(..., "utf-8")`. If `opts.maxRows` is set large, you allocate `rows`, then `cellVals` and `numVals` arrays per column — O(rows × cols) memory and the work runs in a single event-loop tick after the read, blocking the loop. No `await` between read and compute means **the event loop is starved** under concurrent MCP requests. | Stream-parse with `fs.createReadStream(safePath, { encoding: "utf-8" })` and a line/CSV parser (e.g., `csv-parse`). Yield to the loop every N rows with `await new Promise(setImmediate)` or chunk with `for await`. At minimum, `setImmediate` between the read resolution and the per-column mapping. |
| **Medium** | `analyzeData` — `fs.readFile(safePath, "utf-8")` (line ~67) | **No `signal`/cancellation support.** Long-running reads can't be aborted; if the MCP client disconnects, the promise keeps the file descriptor and buffers open. Multiple concurrent reads compound FD pressure. | Add `opts: { workspaceRoot; maxRows?; signal?: AbortSignal }` and pass `signal` to `fs.readFile` (Node 20+ supports `{ signal }`). Reject on abort and close any temp fds. |
| **Low** | `analyzeData` — chart selection (line ~96) | **Non-deterministic ordering under concurrent calls with identical data.** If two callers race and the artifact filename collides (see above), the *first* `writeFile` to land wins; the loser's `chartSvg` is dropped. This is an ordering assumption: callers may assume `artifactPath` reflects *their* chart. | Beyond collision-free names, version the artifact with a content hash (`sha1(chartSvg).slice(0,12)`) so distinct inputs always produce distinct files and the published filename is deterministic. |
| **Low** | `numStats` — `Math.max(...values, 1)` inside `barChartSvg` (line ~35) | **Stack-overflow / OOM on huge arrays.** Spreading >~100k numbers into `Math.max` blows the call stack on some platforms and is a synchronous, non-yielding operation that blocks the event loop — effectively a denial-of-service vector if `maxRows` is misconfigured to a large value. | Use a manual `for` loop (or `reduce`) to find the max, and `await new Promise(setImmediate)` between the per-column reductions to keep the loop responsive. |
| **Low** | `parseCSV` (line ~10) | **No quote-escape handling, but more importantly: synchronous CPU work on the event loop** before any `await`. A large CSV parses entirely synchronously, blocking all other async I/O in the process. | Same fix as above: yield with `setImmediate` every N rows during parsing, or stream-parse. |
| **Info** | Whole module | **No shared mutable state, no event listeners, no `Promise.all` swallowing, no unawaited promises** were found. The module is reentrant-safe *except* for the filesystem-side races listed above. | None required. |

## Summary of recommended hardening

1. **Validate path at open time** (resolve realpath under an open `dirfd`) — fixes the path-traversal TOCTOU.
2. **Atomic, unique artifact writes** — `randomUUID` filename + `tmp` + `rename` — fixes the stampede and torn-write risk.
3. **`AbortSignal` plumbing** — gives callers cancellation and bounds FD/memory lifetime.
4. **Streaming parser + periodic `setImmediate` yields** — eliminates the event-loop starvation under large CSVs / concurrent MCP requests.
5. **Retry-on-`ENOENT` around `mkdir`+`writeFile`** — closes the check-then-act window.

The module does **not** exhibit event-listener leaks, `Promise.all` error-swallowing, or unawaited promise issues; the audit's primary concerns are filesystem TOCTOU and event-loop blocking under load.
