| High | `checkProvider` | **Cache Stampede / Check-Then-Act Race**: The cooldown logic is flawed: if `!status.healthy` and the cooldown hasn't elapsed, `shouldAttemptCheck` is false, but the subsequent `if` only returns early if `status.healthy` is true. Thus, it *always* probes unhealthy providers, completely ignoring the cooldown. Combined with `lastCheck` being updated *after* the `await`, concurrent calls trigger a massive thundering herd of health checks. | Fix the early return condition to `if (!shouldAttemptCheck) return status.healthy;`. | High | `checkProvider` | **Ordering Assumption / Stale Write**: Multiple in-flight `healthChecker` promises (exacerbated by the stampede above) can resolve out of order. An older, slower check returning `unhealthy` could call `markFailed` *after* a newer, faster check called `markHealthy`, overwriting fresh state with stale data. | Serialize health checks per provider using an async mutex (e.g., `async-mutex`), or assign a monotonic sequence ID/timestamp to each check and discard results from outdated sequences. |
| Medium | Class-wide | **Shared Mutable State Without Locks**: The `statuses` Map and `activeProviderId` are mutated across async boundaries without synchronization primitives. While Node.js is single-threaded, the lack of logical locks across `await` points enables the check-then-act and stale write races identified above. | Introduce a per-provider `Mutex` to protect state transitions in `checkProvider`, `markFailed`, and `markHealthy`, ensuring atomic read-modify-write cycles across async boundaries. |
| Low | `markFailed`, `markHealthy` | **Unnecessary `async` / Microtask Deferral**: These methods are marked `async` but contain no `await` expressions.
