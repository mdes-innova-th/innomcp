# Concurrency / Async-Safety Audit — `hotRetriever.ts`

## Context

The module is synchronous in its current form — no `await`, no `Promise`, no event listeners, no I/O. It is a pure normalization layer over already-resolved `toolResult` values. Concurrency issues are therefore confined to **shared mutable state at module scope** and **assumptions made by downstream async callers**.

| # | Severity | Location | Race / Issue | Fix |
|---|----------|----------|--------------|-----|
| 1 | **High** | `let factCounter = 0;` (module scope) + `nextFactId()` (L8–11) | **Shared mutable counter with no synchronization.** Two concurrent `normalize*Facts()` calls running in the same event-loop tick (e.g. inside `Promise.all` over weather + evidence + deterministic pipelines) both invoke `nextFactId(...)`. Because Node is single-threaded this is *atomic per call*, but the IDs are still **non-monotonic with respect to the result objects' logical creation order**, and under a Worker / `Atomics` / future worker_threads port the `++` is a true data race (read-modify-write is not atomic across threads). More importantly: the counter is process-global, so two unrelated requests interleave IDs and leak cardinality across tenants — a correctness/observability bug masquerading as a race. | Replace with a counter owned per call site: pass a `counter` (or a `crypto.randomUUID()`) into the normalizer, e.g. `nextFactId(domain, counter)`; or use `crypto.randomUUID()` for IDs that don't need monotonicity. If a global counter must stay, wrap in an `Atomics.add` on a `SharedArrayBuffer` once this code moves to workers, and namespace by request via a per-request counter object. |
| 2 | **Medium** | `nextFactId("weather")` calls inside `for` loops (L29, L66) | **Check-then-act / TOCTOU between `seen.add(fact.id)` in `mergeRetrievalFacts` (L121) and ID generation in `nextFactId`.** The module guarantees uniqueness only because `factCounter` is a single monotonically-increasing process-global. If `factCounter` is ever reseeded (e.g. after a HMR reload, a forked worker, a test reset, or the fix for #1 that uses per-request counters) two facts can collide and the second will be **silently dropped** by `mergeRetrievalFacts` with no error. This is a swallowed-data race made possible by the dedup in `merge`. | (a) Make IDs collision-free by construction: use `crypto.randomUUID()` or include a per-request nonce. (b) When deduping in `mergeRetrievalFacts`, on collision log/metric `id` and `source.id` — don't drop silently. (c) Validate `id` uniqueness invariant in a test that calls all three normalizers in parallel via `Promise.all`. |
| 3 | **Medium** | `const now = new Date().toISOString();` (L21, L57, L86) and `timestamp: now` | **Wall-clock ordering assumption.** Every fact in a single normalization call shares the same `now`. If the three normalizers run in parallel (the typical hot-RAG pattern: weather ‖ evidence ‖ deterministic), each captures its own `now`, so the *merged* fact array's `timestamp` field is **not globally ordered**, yet downstream `composeFactSummary` / ranking code will sort or compare by it. This is a logical race — concurrent facts may end up ordered opposite to causality, leading to stale-but-newer facts beating fresh-but-older ones. | Pass an explicit `clock: () => Date` (or a fixed `now` string) into each normalizer from the caller so the request has a single coherent timestamp. If ordering matters, use a logical sequence number (the per-request counter from fix #1) and sort by `(seq, timestamp)`. |
| 4 | **Medium** | `extractWeatherEntities(query)` (L139) / `extractISP(query)` (L146) — no concurrency issue per se, but **called from normalize functions that may be invoked concurrently** | **Global regex objects created on every call.** The `provincePattern` and `regionPattern` are re-instantiated on every invocation. In an `async` pipeline this means the V8 regex cache is thrashed and, more relevantly, if these helpers are ever moved into a worker and shared via `postMessage` (structured clone), the shared compiled regex is **not** shared — it gets re-compiled. Not a race today, but a portability hazard that becomes a race the moment regexes are hoisted to module scope and accessed from a worker. | Hoist the compiled `RegExp` objects to module-level `const` (`const PROVINCE_RE = /.../gi;`). Reset `lastIndex` before each use, or use the `.match` string form (which doesn't depend on `lastIndex`) so the same compiled regex is safe under concurrent single-threaded use. Document that the regex is **not** thread-safe if ported to workers. |
| 5 | **Low** | `composeFactSummary` (L128) and `mergeRetrievalFacts` (L114) | **Cache-stampede-prone output shape.** Each call walks and stringifies facts; in a hot async path with many concurrent summarizations, there is no memoization keyed on `(facts reference, length)`. If a higher layer caches by `JSON.stringify(facts)`, two concurrent requests with identical inputs both pay the stringify cost (dogpile, not stampede, but the same family). The functions themselves are pure → trivially memoizable. | Add an optional `WeakMap<RetrievalFact[], string>` cache in `composeFactSummary` keyed on the input array identity, with a size cap (`lru-cache`). This makes concurrent calls with the same fact array O(1) and prevents redundant JSON serialization. |
| 6 | **Low** | `toolResult.result || toolResult.data || toolResult` (L24, L58) | **Truthiness race against streaming/partial results.** If the upstream tool returns a *falsy but valid* `result` (e.g. `0`, `""`, `false`) while `data` is also absent and the toolResult itself is the fallback, the `||` chain falls through to `toolResult` — fine. But if upstream is a streaming tool that emits a *partial* `toolResult` with `result: undefined` and `data: <partial>`, the first read picks `data` (correct), but a **re-read after a microtask** (e.g. a second event-loop turn) may see `result` populated, giving a different shape. The normalizer is called once per result so this is latent — but if any caller does `Promise.race([stream, timeout])` and the winner is a partial, the IDs/timestamps are now based on a snapshot that the timeout-bound caller may treat as authoritative. | Document the contract: `toolResult` must be a fully-resolved, immutable snapshot. Add a `Object.freeze(toolResult)` at the call boundary (in the orchestrator) so any later mutation is a `TypeError` rather than a silent re-read. |
| 7 | **Low** | All `normalize*` exports | **No explicit "no async" contract.** Nothing prevents a future contributor from making these `async` (e.g. to await entity extraction from an LLM). If they do, the caller's `Promise.all([normalizeWeatherFacts(...), normalizeEvidenceFacts(...)])` will interleave `now` captures (see #3) and any `await` between `nextFactId` and the `facts.push` will mean ID generation order ≠ push order — the `seen` set in `mergeRetrievalFacts` will then see IDs out of order but the function is order-dependent in its dedup semantics. | Either (a) freeze the signatures as sync and add `/* sync */` JSDoc + an ESLint rule forbidding `await` inside these files, or (b) commit to async: have each function take a `clock` and a `idGen` so ordering and IDs are explicit. |
| 8 | **Low** | `mergeRetrievalFacts` dedup `seen: Set<string>` (L117) | **Order-dependent merge with no `Promise.all` safety note.** The function is sync and pure, so it has no race of its own. However, the *contract* it implies — "first occurrence wins, by array order" — is fragile under parallel upstream: caller A sees `[[weatherFacts], [evidenceFacts]]`, caller B with a swapped array order sees a different merged result. This isn't a race in the file, but the *reliance* on insertion order in async orchestrators is. | Either sort `merged` by a stable key (`timestamp` from a coherent clock, or `source.id`) before returning, or document the order contract loudly and have the caller pass a pre-sorted array. |
| 9 | **Info** | Whole module | **No event-listener leaks or unawaited promises / `Promise.all` error-swallowing in this file.** The module is a leaf normalizer; it neither registers listeners nor awaits. The audit risk is concentrated in the *callers* (orchestrator that does `Promise.all` over these). Flag for the caller-side review: if the caller does `Promise.all([..., normalizeEvidenceFacts(...)])` without `try/catch` and one rejects, the others' results are dropped by `Promise.all`'s fail-fast — not a bug here, but ensure the caller attaches `.catch` or uses `Promise.allSettled` if partial results are desired. | Add a sibling test in the caller that exercises `Promise.allSettled` and asserts partial results are still merged. |
| 10 | **Info** | `factCounter` reset on process restart | **Not a race, but a documented gotcha worth surfacing:** after a serverless cold start or worker recycle, `factCounter` resets to `0`. Any external system caching fact IDs across restarts (e.g. a dedup index in Redis keyed on `hot:<domain>:<n>`) will see collisions. | Either (a) seed the counter from a persisted source on startup, or (b) migrate to `crypto.randomUUID()` so IDs are globally unique without coordination. Fix #1 already recommends (b). |

## Summary

- **1 High-severity** item (the global `factCounter`): real correctness + cross-tenant data leak risk.
- **3 Medium** items: timestamp drift under parallel calls, silent dedup swallowing collisions, regex-as-shared-state portability hazard.
- **3 Low** items: memoization, snapshot-immutability contract, sync-vs-async contract drift.
- **3 Info** items documenting latent risks for the caller-side code and lifecycle.

The single highest-leverage fix is **#1 + #2 together**: switch from a process-global monotonic counter to `crypto.randomUUID()` (or a per-request counter), which simultaneously removes the cross-tenant ID collision, the silent dedup swallow, and the cross-restart collision class — and makes the module trivially safe under `Promise.all` and under a future worker_threads port.
