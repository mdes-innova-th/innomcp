# Concurrency/Async-Safety Audit — `motherExportService.ts`

This module is **purely synchronous**, so classic async hazards (unawaited promises, `Promise.all` swallowing, event-listener leaks, cache stampedes) don't apply directly. However, it reads from **shared mutable state** (`motherHistory`) without any coordination, which creates several real and latent issues:

| severity | location | race/issue | fix |
|----------|----------|-----------|-----|
| **Medium** | `exportToJSON` / `exportToCSV` — `motherHistory.get(options.limit)` | **Shared mutable state without locks**: `.get()` may return a direct reference to the internal store array. If another call-path mutates `motherHistory` (e.g., appends a run) between the `.get()` call and the completion of iteration/serialization, the exported data is an inconsistent snapshot — partially old, partially new. In the flatMap, this could even cause a provider entry to appear without its parent run if the array is spliced. | Snapshot defensively: `const history = [...motherHistory.get(options.limit)]` (shallow) or deep-clone. Better: have `motherHistory.get()` itself return a frozen copy (`Object.freeze`) so consumers can't mutate and so the snapshot is stable. |
| **Low** | `exportToCSV` — `if (history.length === 0)` | **Check-then-act (benign here)**: The check is on a local binding, so it's safe within a single synchronous tick. However, if `exportToCSV` is ever made async (e.g., to stream rows), the guard becomes a real TOCTOU race. | Document that this function **must remain synchronous**; if made async, re-check length after any `await`. |
| **Medium** | `exportToCSV` — `run.query.replace(…)` / `p.preview.replace(…)` | **Null dereference under concurrent mutation**: If another code-path sets `query` or `preview` to `undefined`/`null` on a live run object (whose reference is shared through `motherHistory`), `.replace()` throws a `TypeError` mid-export. This is a shared-mutable-state side-effect, not a simple null bug. | Guard: `(run.query ?? "").replace(…)` and `(p.preview ?? "").replace(…)`. Better: ensure `MotherRun` objects are frozen or deep-copied on insertion so their fields can't be mutated out from under readers. |
| **Low** | `exportToCSV` — `history.flatMap(run ⇒ run.providers.map(…))` | **Ordering assumptions**: The CSV row order depends on the insertion order of both runs and their `providers` array. If `motherHistory` or `providers` are ever reordered (e.g., sorted in-place by another caller), the export silently reflects the new order with no indication, breaking downstream idempotency expectations. | Sort explicitly before export: `[…history].sort((a,b) => a.timestamp - b.timestamp)` and/or document that output order is not guaranteed to be stable across calls. |
| **Low** | `exportToCSV` — CSV escaping of `query` / `preview` | **CSV injection (not concurrency, but data-integrity)**: Only `"` is escaped; formula prefixes (`=`, `+`, `-`, `@`) in user-controlled `query`/`preview` are not neutralised, allowing macro injection when the CSV is opened in Excel/Sheets. | Prepend a tab or single-quote to cells that start with `=`, `+`, `-`, `@`: `const safe = /^[=+\-@]/.test(val) ? "'" + val : val`. |
| **Info** | `exportToJSON` — `JSON.stringify(history, null, 2)` | **No structural integrity guarantee**: If `motherHistory` is appended to during `stringify`'s traversal (possible if `stringify` somehow yields — it doesn't in current V8, but a custom `toJSON` could), output would be corrupt. | Low risk today, but ensure `MotherRun` types never define `toJSON` that performs async work or mutates sibling entries. |

## Summary

The **core systemic issue** is that `motherHistory.get()` exposes shared mutable state with no copy-on-read or freeze semantics. In Node.js's single-threaded model this is *usually* safe for synchronous consumers, but it creates a fragile contract: any future refactor that makes these functions async, or any `toJSON`/getter side-effect, immediately introduces real data races. The minimal fix is to **snapshot on read** (`[…motherHistory.get(limit)]`) and **null-guard** string fields; the robust fix is to make `motherHistory.get()` return an immutable deep copy.
