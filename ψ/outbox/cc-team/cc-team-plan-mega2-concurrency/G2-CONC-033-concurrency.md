| **Medium** | Per-kind probe implementations (after `buildProbeTargets` definition) | **Cache stampede & duplicate probes**: `runHealthProbes()` is callable multiple times; concurrency is not guarded. If two callers invoke it simultaneously (e.g. during startup and then via an admin route), both will set `"checking"` and fire HTTP requests for every provider. Without in-flight promise caching, this wastes resources and violates rate limits. | Wrap probe logic with a per‑provider `Map<string, Promise<ProviderProbeResult>>` (an “in‑flight” cache). | **Medium** | Per-kind probe implementations | **Check‑then‑act race on `probeStatus`**: The probe sets `status: "checking"` unconditionally before the HTTP call. If another caller sees `"checking"` and decides to wait/retry, it may read an intermediate `"online"`/`"offline"` that is immediately overwritten back to `"checking"` by the second probe start, yielding inconsistent reads. | Atomically set `"checking"` only if the current status is not already `"online"` or `"offline"` (or use the in‑flight promise cache to skip probing entirely when a probe is active). |
| **Low** | Exported `probeStatus` map | **Shared mutable state without locks**: The `Map` is read by `agentLeaderboard` (or other consumers) while it is being mutated by probe completion callbacks. In Node.js’s single‑threaded event loop this cannot cause data corruption, but reading a partially written object is impossible because object references are assigned atomically. However, the lack of a well‑defined snapshot point can still cause a consumer to see a transient `"checking"` or miss a status update on a fast refresh. | No change required for correctness, but consider returning a deeply‑frozen snapshot (e.g. `Object.freeze({...})`) or a promise-based reactive store if clients need a consistent view. |
| **Low** | Probe timeout handling (likely inside per‑kind implementations) | **Delayed timer leak**: Using `setTimeout(() => controller.abort(), 5000)` without clearing the timer when the fetch completes normally means the timer will fire needlessly (though harmless).
