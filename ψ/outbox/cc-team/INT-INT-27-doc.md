## Root Cause Analysis
**Bulk LLM generation without a build gate committed directly to main.**  
The engineer intended to accelerate development by using an LLM to generate boilerplate code, but the output was pushed straight to `main` without any review or automated validation. The repository’s main branch allowed direct pushes because branch protection rules were never fully enforced for a legacy service account. No pre‑receive hook, size limit, or content pattern gate existed to block such a large, auto‑generated change. The team lacked a documented policy for integrating AI‑generated code.

_การวิเคราะห์สาเหตุราก:_ การสร้างโค้ดจำนวนมากด้วย LLM โดยไม่มี gate ก่อนขึ้น main วิศวกร push ผลลัพธ์จาก LLM เข้า main โดยตรงผ่าน service account ที่ยังมีสิทธิ์เขียน เพราะ branch protection ไม่ครอบคลุม account นั้น ไม่มี hook ตรวจสอบขนาดหรือรูปแบบเนื้อหา และทีมไม่มีนโยบายการใช้ AI สร้างโค้ด

## 5 Contributing Factors
1. **Missing branch protection** – `main` allowed direct pushes from a service account that was omitted from the rule set.
2. **Over‑privileged service account** – `svc-mega-bot` retained legacy write access without any restriction.
3. **No automated anomaly detection** – No pre‑receive hook to reject commits with >500 files or >10,000 line changes.
4. **Lack of AI‑assistance guidelines** – Engineers had no documented process for reviewing and landing LLM‑generated code.
5. **Insufficient monitoring of direct pushes** – Alerts existed only for CI failures, not for unexpected push patterns.

_5 ปัจจัยร่วม:_ 1) ขาด branch protection สำหรับ service account 2) service account มีสิทธิ์เขียนเกินจำเป็น 3) ไม่มีระบบตรวจจับ commit ที่ผิดปกติ 4) ไม่มีแนวทางปฏิบัติสำหรับการใช้ AI สร้างโค้ด 5) ไม่มีระบบเฝ้าระวังการ push ตรงเข้า main

## 5 Action Items (Phases 1–5)
- **Phase 1 (Immediate)** – Enforce strict branch protection on `main`: require a pull request, require status checks to pass, and disable force‑push and direct push for *all* accounts, including service accounts.
- **Phase 2 (Week 1)** – Implement a pre‑receive hook that rejects any commit changing more than 500 files or 10,000 lines unless the commit message contains an `--allow-large` token and the author is in an exception list.
- **Phase 3 (Week 3)** – Integrate a CI gate that scans diffs for typical LLM patterns (e.g., `generated by`, repetitive boilerplate) and blocks the build unless a “LLM-approved” label is present on the PR.
- **Phase 4 (Week 4)** – Revoke direct push permissions for all human and service accounts; allow writes only through the PR merge flow with automated checks.
- **Phase 5 (Week 6)** – Conduct a mandatory workshop on safe AI‑assisted development and update the contributor guide to mandate that LLM‑generated code must be authored on a feature branch, reviewed, and explicitly tagged.

## 5 Lessons Learned
1. **Automation gates are irreplaceable** – Human mistakes on critical branches can only be blocked by automated policy enforcement.
2. **Service accounts demand least privilege** – Even internal bots must not have direct write access to main.
3. **Bulk generation hides defects** – LLM output must flow through the same review and CI pipeline as any other code change.
4. **Real‑time push monitoring saves time** – If direct‑push alerts existed, detection would have been minutes instead of hours.
5. **Culture must adapt to AI** – Tooling and processes must evolve to balance AI‑accelerated development with robust safety nets.

_5 บทเรียนที่ได้รับ:_ 1) automation gate คือสิ่งจำเป็น 2) service account ต้องใช้สิทธิ์น้อยที่สุด 3) โค้ดที่สร้างโดย AI ต้องผ่าน review เหมือนโค้ดปกติ 4) การเฝ้าระวัง push โดยตรงช่วยลดเวลาเสียหาย 5) วัฒนธรรมและเครื่องมือต้องปรับตัวให้ทันกับยุค AI
