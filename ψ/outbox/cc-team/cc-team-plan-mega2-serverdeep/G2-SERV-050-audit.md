# Deep Bug Audit: `innomcp-server-node/src/memory/types.ts`

This is a type-definition file, so direct runtime bugs (unhandled rejections, leaks, etc.) cannot exist here. However, the contracts declared in this file directly cause or permit bugs in any implementation that conforms to them. The audit below treats this as an **interface contract audit** and identifies how the type definitions themselves are the root cause of downstream issues.

| Severity | Location | Issue | Fix |
|---|---|---|---|
| **Critical** | `EmbeddingService.embed()` return type `Promise<number[] \| null>` | Returning `null` makes the failure mode indistinguishable from a successful empty result. Callers that propagate `null` into `VectorStore.add()` will either silently store items with no embedding (breaking vector search) or throw an unhandled rejection deep in the store. There is no protocol to signal *why* embedding failed (rate limit, OOM, model unavailable), so retry/fallback logic is impossible to implement correctly. | Change to `Promise<number[]>` and reject the promise with a typed `EmbeddingError` (e.g. `{ code: 'RATE_LIMIT' \| 'MODEL_DOWN' \| 'INVALID_INPUT', cause }`). Reserve `null` only if you genuinely support "embedding not applicable for this content," and expose that as a separate `embedOptional(): Promise<number[] \| undefined>`. |
| **Critical** | `VectorStore.search()` | No documented contract for behavior on an empty/mismatched-dimension `queryVector`. Implementations will either throw `DimensionMismatchError` (unhandled rejection if caller doesn't catch) or silently return empty results. Race condition risk: if `add()` is in-flight when `search()` runs against an in-memory index, results may be inconsistent. | Define explicit error type `VectorStoreError` with variants `DIMENSION_MISMATCH`, `NOT_READY`, `IO_ERROR`. Add `await store.ready` (or an explicit `initialize()` step) so callers cannot `search()` before the index is loaded. Document that `search()` snapshots the index at call time. |
| **High** | `VectorStore.add()` doc comment: *"If item has no embedding, store implementation might generate it or throw."* | The "might" makes behavior non-deterministic. If the store *does* generate embeddings, it secretly depends on an `EmbeddingService` (tight coupling, not visible in the type). If it *throws*, that throw is an unhandled rejection unless the caller wraps every `add()`. Two implementations of the same interface can behave completely differently. | Split into two methods: `add(item: MemoryItem): Promise<void>` requires `item.embedding` to be present (throws typed `MissingEmbeddingError` otherwise); `addWithEmbedding(text, metadata): Promise<MemoryItem>` is the convenience method that calls the embedding service. Remove the ambiguity. |
| **High** | `VectorStore.search()` and `searchByKeyword()` return `MemoryItem[]` with no pagination, no total count, no cursor | Large result sets will allocate huge arrays (memory leak / OOM risk) and are transmitted in one MCP response (potential payload size violation). No way to implement incremental/streaming results. | Change return type to `Promise<SearchResult>` where `SearchResult = { items: MemoryItem[]; total: number; nextCursor?: string }`. Enforce a server-side `MAX_LIMIT` (e.g. 100) to bound memory. |
| **High** | `search(queryVector, limit?)` and `searchByKeyword(query, limit?)` | `limit` is optional with no default or max. A caller passing `undefined` may get unbounded results (DoS / OOM). A caller passing `Number.MAX_SAFE_INTEGER` causes the same. | Make `limit` required with documented min/max (`1..100`). Or keep optional but document the default and enforce a hard ceiling inside the interface contract (e.g. via a static `VectorStore.MAX_LIMIT`). |
| **High** | `save()` / `load()` — *"if applicable"* | Implementations that are in-memory-only must "throw or no-op" — but nothing in the contract says which. Callers cannot know whether persistence succeeded. A silent no-op after a process restart silently loses all memory (data loss bug). | Separate the interface: `PersistedVectorStore extends VectorStore { save(): Promise<void>; load(): Promise<void>; }`. In-memory stores should not implement this. Add a `isPersistent(): boolean` capability flag, or use runtime capability discovery, so callers can refuse to use a non-persistent store for durable memory. |
| **High** | `MemoryItem.metadata?: Record<string, any>` | Untyped metadata propagates untrusted data into MCP tool results. Since MCP serializes via JSON, `undefined`, functions, symbols, `BigInt` (throws on serialize), circular references (throws), and very large strings (DoS) are all possible. No validation layer exists. | Define a strict metadata schema per `MemoryType` (discriminated union), e.g. `FileMemoryMetadata = { source: string; size: number; mime: string }`. Validate at construction with a schema library (Zod) and reject `undefined`/non-JSON-safe values. |
| **High** | `embedding?: number[]` (no dimension constraint) | Nothing prevents an item with a 768-dim embedding being mixed with 1536-dim embeddings in the same store. `search()` will then either throw at runtime or return garbage. This is a latent dimension-mismatch race / corruption bug. | Either (a) make `VectorStore` generic in the dimension: `VectorStore<Dim extends number>`, or (b) add `embeddingDim: number` to the store constructor and validate on `add()`. The `search()` signature should also carry the dimension: `search(query: Vector<Dim>, …)`. |
| **Medium** | `MemoryItem.timestamp: number` | Ambiguous unit (ms vs s). If mixed, time-based filtering and dedup logic become wrong. No timezone, no monotonicity guarantee. | Use `timestamp: number` documented as **Unix epoch milliseconds** (and a `const _unit: 'ms'` brand), or switch to `string` ISO-8601 / `Date` object. |
| **Medium** | `MemoryItem.id: string` | No format/version constraint. MCP requests referencing stale IDs across restarts will fail silently if `load()` regenerates IDs. No namespacing, so collisions across `MemoryType`s are possible in a shared store. | Use a branded type: `type MemoryId = string & { readonly __brand: 'MemoryId' }`. Prefix by type, e.g. `file_${ulid()}`, `tool_${ulid()}`. Validate on `add()` (reject duplicate IDs with a typed error). |
| **Medium** | `EmbeddingService.embed(text: string)` | No `AbortSignal`, no timeout. A hanging embedding service call will leak the MCP request's resources (sockets, file handles) and tie up the request slot indefinitely — a classic server-node resource leak. The "returns null on failure" design also means a hang is indistinguishable from a slow response, masking the leak. | Add a second parameter: `embed(text: string, options?: { signal?: AbortSignal; timeoutMs?: number }): Promise<number[]>`. The implementation contract must enforce `timeoutMs` (reject with `TimeoutError`) and honor `signal` (reject with `AbortError`). Drop the `\| null` so a hang surfaces as an error, not silent success. |
| **Medium** | `VectorStore.add()` concurrency | If two MCP requests call `add()` concurrently, in-memory stores that mutate an array will lose data (no synchronization contract). File-backed stores will interleave writes and corrupt the file. | Document thread-safety expectations explicitly. Either (a) require `add()` to be internally serialized with a mutex/queue, or (b) add `batchAdd(items: MemoryItem[]): Promise<void>` and `disallow` parallel `add()` calls. For file-backed stores, mandate atomic write (write to `tmp`, fsync, rename). |
| **Medium** | `VectorStore.searchByKeyword()` | No language/tokenization contract, no stop-word handling, no case-folding guarantee. Two implementations of the same interface will return different results for the same query — a contract bug that breaks caller caching and testability. | Either narrow the interface to a specific tokenizer contract (e.g. "lowercase, Unicode NFKC, split on `/\s+/`, drop stop words from a fixed list") or split into `searchByKeyword(query, options: { tokenizer: Tokenizer })`. Provide a reference tokenizer. |
| **Low** | `MemoryType = "file" \| "tool_result" \| "db_row" \| "conversation"` | Closed string union, which is good, but downstream code will likely do `switch(item.type)` and forget a case. With `noUncheckedIndexedAccess`/exhaustive checks off, the compiler won't catch the missing branch, leading to silent fallthrough bugs. | Add an exhaustiveness helper: `const _exhaustive: never = item.type;` in a default branch of expected switch sites, or export `assertNever(item.type)`. |
| **Low** | `EmbeddingService` and `VectorStore` are not branded/distinguished from plain objects | Nothing prevents a caller from passing `{ embed: () => Promise.resolve([]) }` as a fake. In MCP servers this enables accidental mocking in production and confusing errors. | Use TypeScript branding (`interface` is fine, but consider `readonly` properties and a `readonly __nominal?: unique symbol` brand) and validate with `zod` at the trust boundary (e.g. when constructing a server). |
| **Low** | `MemoryItem.content: string` | No size limit. An embedding call on a 100 MB string will OOM the server. A vector store entry on a 100 MB string will OOM the JSON serializer. | Add `readonly contentByteLimit: number` (e.g. 64 KiB) to the embedding service and vector store, reject oversized content with a typed error. Document the limit in JSDoc. |
| **Info** | `VectorStore.save()` / `load()` — return `void` | No way to know what was saved, what version, or what was skipped on load. Hard to detect partial corruption / migration needs. | Return a `PersistenceReport = { written: number; skipped: number; version: number; errors: PersistenceError[] }`. |
| **Info** | No error types exported from this module | Callers will `catch (e: any)` and lose structure. MCP error envelopes will be inconsistent. | Export a `MemoryError` union (e.g. `EmbeddingTimeoutError`, `DimensionMismatchError`, `MissingEmbeddingError`, `PersistenceError`, `CapacityExceededError`) and document that all `VectorStore` / `EmbeddingService` rejections must be one of these. |

## Summary of Highest-Impact Contract Bugs

1. **`null` vs throw in `embed()`** — masks failures as success, the single worst pattern for a memory system. (Critical)
2. **Ambiguous `add()` embedding behavior** — "might generate or throw" is a recipe for unhandled rejections in production. (High)
3. **No timeout / AbortSignal on `embed()`** — guaranteed resource leak in server-node context. (Medium-High)
4. **No dimension typing on embeddings** — latent corruption in mixed-dimension stores. (High)
5. **Untyped `metadata`** — DoS / DoS-by-serialization / data integrity bug surface for MCP responses. (High)

The file itself is small, but it is the **root cause** of most of the runtime bugs the audit was asked to find, because every implementation will inherit these ambiguities.
