| Severity | Location | Issue | Fix |

| High | `catch` block in `execute`, after `if (isDbMissing) { ... }` | The `catch` block only returns a stub response when the DB is missing. For all other database errors (e.g., syntax, connection), execution falls off the end of the function without a return, yielding an implicit `undefined` result. This breaks the MCP content requirement and can cause a server-side error or a protocol violation. | Ensure the catch block always returns an error envelope. Add an `else` or default return that produces an `errorResult` with code `DB_ERROR` and the error message. |
| High | `execute` → `await query(sql, params)` | No timeout is applied to the database query. If the database hangs or takes very long, the tool execution blocks indefinitely, consuming server resources and failing the client on timeout. | Introduce a configurable timeout using `Promise.race` against a timeout promise, or configure the `query` utility with a per-call timeout option. Wrap the query call to reject after a reasonable duration (e.g., 10 s). |
| High | `errorResult()` helper and all call sites that return an error | MCP tool results should signal an error to the client by setting `isError: true` in the result object (or by throwing an `McpError`). The current `errorResult` returns only `{ content: … }` without `isError`, so the server treats the response as a successful execution. Clients relying on the protocol will interpret all responses as success and must manually parse the JSON to detect errors. | Modify `errorResult` to return `{ content: …, isError: true }` or throw an `McpError` with the appropriate code. Update all error returns accordingly. |
| Medium | `execute` → `catch` block, stub fallback | The stub fallback does not use the confidence mechanism at all – it returns stub entities without checking `confidence_required` or filtering by confidence.
