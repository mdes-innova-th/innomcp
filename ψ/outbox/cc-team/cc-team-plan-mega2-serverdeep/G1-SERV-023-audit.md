| **Critical** | `execute` / `filePath` | **Path traversal (Arbitrary File Read):** `filePath` is passed unsanitized to `fs.readFile`, `XLSX.readFile`, and `path.extname`. Attackers can supply paths like `../../../etc/passwd` to read any file the server process can access. | Sanitize `filePath` with `path.resolve` against a strict workspace root / allowlist; reject absolute paths, parent directory references (`..`), and symlinks outside the root. |
| **Critical** | `readPDF`, `readExcel`, `readWord` | **Missing timeouts / DoS:** PDF, Excel, and Word parsers have no execution timeout. A malformed or pathological file can cause infinite loops or indefinite hangs, freezing the MCP server node and breaking all concurrent sessions. | Wrap every parser call in `Promise.race` with a `setTimeout` that rejects after a configurable limit (e.g., 30s). Use `AbortController` / signals where libraries support them. |
| **High** | `execute` catch block | **MCP error envelope violation:** Errors are swallowed and returned as a `text` content blob with `success: false`. Per MCP protocol, this signals a *successful* tool execution to the client; the client cannot distinguish failure or retry. | Throw a proper error (`throw new McpError(ErrorCode.InternalError, message)` or standard `Error`) so the MCP SDK returns a JSON-RPC error response with the correct `error` envelope. |
| **High** | `readExcel` helper | **Event-loop blocking:** `XLSX.readFile()` is synchronous. Large `.xlsx` files will block the Node.js event loop, stalling all other MCP requests and background tasks until parsing completes. | Asynchronously read the file into a `Buffer` (`fs.readFile`), then use `XLSX.read(buffer)`. For large files, offload to a `Worker` thread. |
| **High** | `execute` / `filePath` | **No file size limits / OOM risk:** The entire file is loaded into memory (`Buffer` → parsed object → `JSON.stringify`), with no `fs.stat` guard. A multi-GB file will crash the process with an out-of-memory error. | Check `fs.stat(filePath).size` against a `MAX_FILE_SIZE` limit before reading; stream or chunk large outputs; truncate parsed content before serialization. |
| **High** | `readPDF` helper | **Invalid API usage / silent failure:** `pdf-parse` does not natively support a `max` option. The `maxPages` parameter from user input is silently ignored, violating user expectations and potentially returning massive PDFs. | Remove the invalid option; implement page limiting by parsing the PDF catalog or post-truncate the text output if the library doesn't support it. |
| **Medium** | Schema / `filePath` | **Base64 schema lie:** The schema description states `filePath` can be "Path หรือ base64", but the code always treats it as a filesystem path (`fs.readFile`). Passing base64 will cause a file-not-found error or path traversal attempt. | Detect base64 input (e.g., via regex `^data:` or `^[A-Za-z0-9+/]+={0,2}$`), decode to `Buffer`, and pass the buffer directly to the parsers; branch before `fs.readFile`. |
| **Medium** | `readWord` / schema | **Dead option:** `options.includeImages` is declared in the Zod schema but `mammoth.extractRawText` cannot extract images; the flag is silently ignored. | Remove the option from the schema, or switch to `mammoth.convertToHtml` with an `img` handler if image extraction is required. |
| **Medium** | `execute` / `auto` | **Broken auto-detection for base64:** If a user passes a base64 string with `fileType: "auto"`, `path.extname` is called on the base64 payload, yielding garbage or `""`, causing a confusing "unsupported type" error. | Branch on base64 vs. path *before* extension detection; for base64, require an explicit `fileType` or sniff the buffer magic bytes. |
