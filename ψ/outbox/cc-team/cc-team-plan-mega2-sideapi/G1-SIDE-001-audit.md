| Severity | Location | Issue | Fix |
|----------|----------|-------|-----|
| **Medium** | `healthCheck()` / `getMode()` | The `_mode` variable is only set to `"live"` inside `healthCheck()`. If `healthCheck()` is never called, or if it fails temporarily but the pool later works, the system will incorrectly report `"scaffold"` in `getMode()` while queries succeed. This breaks feature‑gating logic that relies on the mode. | Update `_mode` to `"live"` inside `getPool()` (or during the first successful query) so that the mode reflects actual database availability. Use a try/catch around a `SELECT 1` in `getPool` or in a dedicated initialisation function. |
| **Low** | `getPool()` – `mysql.createPool()` call | `mysql.createPool()` is called outside any try/catch block. If the configuration contains invalid values (e.g., malformed host, incorrect port type) the synchronous throw will crash the application or propagate unhandled. | Wrap the pool creation in a try/catch. Log the error and either return `null` or rethrow a custom `DatabaseConfigurationError`. |
| **Medium** | `getPool()` – pool options | `queueLimit: 0` disables queue length limiting. Under sustained load, connection requests can pile up indefinitely, leading to memory exhaustion and denial‑of‑service. | Set a sensible `queueLimit` (e.g., 100–1000) to reject excess requests with a “too many connections” error instead of consuming unbounded memory. |
| **Low** | `query(sql, params)` | The function passes the raw `sql` string directly to the driver. There is no enforcement that parameters are used for dynamic values. If a caller constructs SQL via string concatenation with user‑supplied data, SQL injection is possible. | Document clearly that all dynamic values **must** be passed via the `params` array. Optionally implement a runtime assertion that `sql` does not contain unescaped user input (e.g., disallow `?` if `params` is missing, though this is imperfect). |
| **Info** | `isConfigured()` / `getPool()` | Database credentials are read directly from environment variables without any validation or encryption. Secrets may be exposed in logs or process dumps. | Validate credential format (non‑empty, correct length) before use. Integrate a secrets manager (e.g., HashiCorp Vault, AWS Secrets Manager) instead of plain env vars in production. |
| **Low** | Module – no pool cleanup | No function is exported to close the connection pool (e.g., `pool.end()`). This can cause connections to remain open on graceful shutdown, leading to idle database connections and port exhaustion. | Add a `closePool()` function that calls `pool.end()` and reset the module state. Call it from the application’s shutdown handler. |
| **Low** | `isDetectBridge()` defined but unused | The detect‑bridge mode flag is never consulted in any query or logic. The module always queries normal `db_aces` tables; bridge‑specific tables (`nip`, `record`) are not used, making the feature non‑functional. | Either implement the bridge path inside `query()` (or a separate `queryBridge()` function) that switches table names and queries based on `isDetectBridge()`, or remove the dead code to avoid confusion. |
