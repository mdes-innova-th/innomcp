## API Contract Audit — `mdesModels.ts`

| Severity | Endpoint | Issue | Fix |
|----------|----------|-------|-----|
| 🔴 **High** | `GET /api/mdes/models/:name` | No validation on `:name` param. Allows empty/whitespace strings, overly long strings, or values with special characters that may break downstream SQL/NoSQL queries, log injection, or cache keys. Risk of injection, log forging, and cache poisoning. | Add input validation: trim, enforce `^[A-Za-z0-9._-]{1,128}$` (or a documented pattern), reject otherwise with `400`. Use a validator (e.g., `zod`, `express-validator`). |
| 🔴 **High** | `GET /api/mdes/best/:task` | Unsafe type assertion (`as Task`) — `req.params.task` is typed as `string` but is cast without runtime guarantee. Although the `includes` check follows, the cast is a code-smell that will break if someone reorders logic later. Also, no length/encoding check on the param before validation. | Remove the `as Task` cast; assign to a plain `string` and narrow it only after `validTasks.includes(...)` via a type guard. Reject values longer than e.g. 64 chars early. |
| 🟠 **Medium** | All endpoints | Inconsistent response envelope: `GET /models` and `GET /models/:name` return `{ success, data }`, but `GET /best/:task` returns `{ success, model }`. Clients cannot reliably parse. Breaking change for any consumer normalizing on a `data` field. | Pick one envelope and apply it everywhere. Recommend `{ success: boolean, data: T }` consistently, or document both shapes. |
| 🟠 **Medium** | All endpoints | Error response shape is inconsistent: success uses `{ success, data }`, errors use `{ error: string }` with no `success` flag, no `code`/`requestId` field. Hard for clients to programmatically branch on error type. | Standardize to `{ success: false, error: { code, message, requestId? } }`. Include a stable `code` (e.g., `MODEL_NOT_FOUND`, `INVALID_TASK`, `INTERNAL_ERROR`). |
| 🟠 **Medium** | `GET /api/mdes/best/:task` | No `404` case when no model qualifies for a valid task. Returns `200` with possibly `null`/empty `model`, ambiguous for clients. | If `getBestModelForTask` returns `null`/undefined, respond `404` with `{ error: { code: "NO_MODEL_FOR_TASK", message: ... } }`. |
| 🟠 **Medium** | `GET /api/mdes/models/:name` | Throws internal error messages to client (`err.message` echoed in `500`). Risks leaking stack/DB/cache internals (e.g., file paths, driver names). Information disclosure. | Log the full error server-side with a `requestId`; return a generic `"Internal error"` (or localized safe message) to the client. |
| 🟠 **Medium** | `GET /api/mdes/models` | No pagination, no filtering, no `limit`/`cursor` params. If the model list grows (hundreds/thousands), the response becomes unbounded — a DoS/performance risk and a breaking change once pagination is added. | Add `?limit` (default 50, max 500) and `?cursor` now, even if a single-page response is currently returned. Document in OpenAPI. |
| 🟠 **Medium** | `GET /api/mdes/models/:name` | No rate limiting / abuse protection. Unbounded reads against cache; combined with the path-traversal-adjacent risk above, it's a soft target. | Add rate limiting middleware (e.g., `express-rate-limit`) per IP and per route. |
| 🟡 **Low** | All endpoints | No `Content-Type` / charset guarantees; no `Cache-Control` headers. `GET /models` in particular is a strong candidate for `Cache-Control: public, max-age=...` and ETag support. | Set explicit `Cache-Control` (e.g., `public, max-age=60` for list, `max-age=300` for specific model) and emit `ETag` based on cache version. |
| 🟡 **Low** | `GET /api/mdes/best/:task` | `validTasks` list duplicated in code and only documented in a JSDoc comment. Undocumented in OpenAPI/Swagger. Adding a new task silently is a breaking change for anyone switching on the value. | Externalize the task list (e.g., a shared `const TASKS = [...] as const` exported from a `tasks` module) and generate/publish OpenAPI. |
| 🟡 **Low** | `GET /api/mdes/models/:name` | Path parameter `:name` is case-sensitive but model names often aren't. `Foo` and `foo` will return different results — a likely footgun. | Decide on a case policy (recommend case-insensitive lookup, or document the sensitivity explicitly). |
| 🟡 **Low** | `GET /api/mdes/best/:task` | Same case-sensitivity ambiguity for `task`. `Thai` vs `thai` — `includes` will reject. | Normalize via `req.params.task.toLowerCase()` before validation, or document. |
| 🟡 **Low** | All endpoints | No request ID propagation. Errors cannot be correlated with server logs. | Add `requestId` middleware (e.g., `express-request-id`) and include it in every response (`X-Request-Id` header + `error.requestId`). |
| 🟡 **Low** | All endpoints | No `OPTIONS` / CORS preflight handling visible. The router doesn't declare allowed methods/headers — may break browser clients. | Either mount `cors()` middleware or add an explicit `router.options('*', ...)` handler. |
| 🟡 **Low** | `GET /api/mdes/models` and `GET /api/mdes/models/:name` | No `If-None-Match` / `304 Not Modified` support despite data being cache-backed. Minor but useful. | Emit `ETag` on `200`; on conditional request, respond `304` with empty body. |
| 🟢 **Info** | `GET /api/mdes/models/:name` | Missing `410 Gone` semantics consideration: if a model is intentionally removed, clients get `404` forever; a `410` with a `Sunset` header would be more correct for retired models. | Optional: return `410` with `Sunset` header when a model is explicitly retired (vs. never existed → `404`). |
| 🟢 **Info** | `GET /api/mdes/best/:task` | No deprecation/availability header for the "best" model — clients can't tell how fresh the recommendation is. | Add `X-Cache-Generated-At` or similar so callers know the staleness of the "best" answer. |
| 🟢 **Info** | All endpoints | No OpenAPI/Swagger doc generated from the route. `task` enum and `:name` format are only in comments. | Add `@openapi` JSDoc annotations (or `zod-to-openapi` schemas) and serve `/docs`. |

### Top 5 prioritized actions
1. **Validate `:name` and `:task`** with a schema (zod/joi) — eliminates the type-assertion smell and the injection-adjacent risks.
2. **Unify the response envelope** (`{ success, data }` + `{ success:false, error:{code,message} }`) across all three routes — single breaking-change if not fixed before external consumers ship.
3. **Stop leaking `err.message` to clients**; add a request-ID and server-side logging instead.
4. **Add `404` for "no best model for valid task"** — currently ambiguous.
5. **Introduce pagination + `Cache-Control`/`ETag` on `GET /models`** before the list grows and forces a harder breaking change.
