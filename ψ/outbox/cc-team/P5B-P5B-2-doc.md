# RECOVERY-SUMMARY.md

## 1. Incident Summary  ·  สรุปเหตุการณ์

On 2025-03-12 the automated pipeline **MEGA‑100** executed a bulk LLM‑driven code‑generation task across the `innomcp-node` repository. The run introduced widespread regressions that broke the existing codebase:

- **14 TypeScript compilation errors** – missing imports, incorrect function signatures, and orphaned type references.
- **5 files quarantine‑flagged** – LLM‑hallucinated modules with no real counterpart in the project.
- **3 critical API routes** silently broken – endpoints returned 500 or wrong payloads because they were rewritten against non‑existent internal services.
- **Safety‑guard removal** – `@ts-nocheck` was injected into 6 files, masking all type violations and delaying detection.

Root cause: the LLM optimised for syntactic fluency but lacked the domain‑specific invariants of `innomcp-node` (custom middleware chains, strict TypeScript strictness, and real‑API contracts). It deleted guard clauses, rewired imports to false modules, and assumed a flat utility layer that did not exist.

## 2. Phase 1 Fixes · การแก้ไขระยะที่ 1

**Goal:** Restore core stability and eliminate immediate compiler noise.

- **4 core services restored to last‑known‑good commits.**

- **14 residual `tsc` errors fixed.**

- **5 hallucinated files quarantined.**  
  Files such as `src/ai-predictor.ts`, `lib/magic-cache.ts` and three others were moved to a `/quarantine/` directory. They contained logic that referenced non‑existent databases and APIs, posing a risk if merged.

## 6. Phase 5: Pre‑commit Hooks · ระยะที่ 5: ติดตั้ง pre‑commit hooks

To prevent automated tools from introducing unsafe changes, a pre‑commit guard was installed:

- **Husky** + **lint‑staged** configured.  
  On every commit, the following run in order:
  1. `prettier --check` (code style)
  2. `eslint --max-warnings 0` (lint rules, including `no-ts-nocheck`)
  3. `tsc --noEmit` (full type‑check)
  4. `vitest --run` (unit/integration tests related to changed files)

If any step fails, the commit is rejected. The configuration file `.husky/pre-commit` is version‑controlled and enforced on all contributor machines.

## 7. Lessons Learned  ·  บทเรียนที่ได้รับ

1. **Never trust LLM‑generated types without a strict lint gate.**  
   The MEGA‑100 pipeline should have been forced through the same type‑checking CI stage as human code. Its changes bypassed `tsc` because of `@ts-nocheck` – an anti‑pattern that must be forbidden via lint rules.
2. **Isolate LLM output in a dedicated branch with mandatory human review.**  
   Bulk generation can alter dozens of files silently. A strict merge‑request policy (minimum one reviewer, no direct push) would have caught the hallucinated modules before they reached `main`.
3. **Automatic code generation must include real‑API contract validation.**  
   The three broken routes could have been detected immediately by spinning up a lightweight integration‑test harness that checks response schemas. Mock servers must mirror production signatures, not be invented by the LLM.
4. **Runtime observability flags must survive generation.**  
   Health endpoints, WS reconnection signals, and fast‑path guards are non‑functional requirements that LLMs easily strip. Codify them as project‑wide rules or auto‑tests so their removal triggers a build failure.
5. **Pre‑commit hooks are the last line of defence.**  
   Phase 5’s hooks block untested, untyped, and unformatted code at the developer’s machine. They would have prevented the original broken commit from entering the repository. Invest in a robust local‑check pipeline alongside CI.
