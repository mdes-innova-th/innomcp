# LESSONS LEARNED: The MEGA-100 Incident

> **Context:** On 2025-01-15, our team attempted to generate 100 production-bound backend services in a single autonomous run using an LLM agent. By hour 6, the system had hallucinated 4 non-existent API endpoints, duplicated 3 service files with divergent schemas, silently dropped 11 error handlers, and committed 2 files containing placeholder `TODO` text into what the agent labeled as "production-ready." This document captures the engineering lessons we paid for in incident reports.

---

## Lesson 1: Bulk Generation Hides Compounding Errors

**Before:** "Let's parallelize 100 service scaffolds at once — we'll finish by lunch!"
**After:** Cap concurrent LLM generation to **5 tasks per worker**, with a mandatory per-task diff review before the next batch.

Each generated file looked plausible in isolation. Across 100 files, small inconsistencies (auth header casing, timestamp formats, error response shapes) compounded into a system that couldn't pass contract tests. We now run a `lint-and-contract-validate` hook after every Nth file.

---

## Lesson 2: The "Done" Signal Is a Lie

**Before:** Agent reported `STATUS: COMPLETE — 100/100 services ready for deployment.`
**After:** Completion is defined as **"all CI checks green on a clean checkout,"** not agent self-report.

The agent's self-assessment is generated by the same process that may have introduced the error. We treat agent output the same way we treat a junior dev saying "tests pass" without showing the test runner. Our CI now fails any PR that doesn't include a verifiable green build artifact.

---

## Lesson 3: Schemas Must Be Exogenous, Not Generated

**Before:** "Let the LLM design the Postgres schema for each service based on the feature description."
**After:** Schemas, API contracts, and shared types live in a **frozen, human-authored repo** that the agent reads but never modifies.

The MEGA-100 run produced 14 subtly different `users` table definitions. When services tried to share a database, nothing joined. We now generate *implementations* of contracts, not contracts themselves.

---

## Lesson 4: Placeholder Detection Is a First-Class Check

**Before:** Code review was the only line of defense against `TODO`, `FIXME`, `NotImplementedError`, and `pass` placeholders.
**After:** A pre-commit hook fails any file containing forbidden tokens unless the path matches `*/scratch/*` or `*/experiments/*`.

Two services deployed with `raise NotImplementedError("TODO: implement billing")` in critical paths. Our CI now treats these tokens as **build-breaking errors** for any `src/` path. Catch rate in our first week: 47 occurrences in generated code.

---

## Lesson 5: Dry-Run, Smoke-Test, *Then* Trust

**Before:** "The code compiled and unit tests passed, ship it."
**After:** Every generated service must pass a **containerized end-to-end smoke test** that boots the binary, hits `/healthz`, and executes one happy-path call against a live dependency.

LLM-generated code can pass static analysis and unit tests while being functionally broken at runtime (wrong env var names, missing migrations, broken connection strings). The smoke test caught 9 of 100 services on first run after this change.

---

## The Meta-Lesson

The MEGA-100 incident wasn't caused by the LLM being "wrong." It was caused by us treating **generation as the bottleneck** when **verification** was the actual bottleneck. We had built a fast factory with no quality control. The fix was not a better prompt — it was a better factory.

> *Rule of thumb: If you can't afford to review 100 PRs, you can't afford to generate 100 files.*
