# Phase 8.4: Weather Resilience Acceptance Criteria

## 1. Full-Stack Abort Chain

- **Requirement:** The `AbortSignal` must successfully traverse the `mcpClient` boundary.
- **Proof:** To achieve this, the MCP Server tool definitions for TMD fetching MUST include an explicitly bound schema (e.g. `EmptyArgsSchema` or equivalent Zod schema). If `inputSchema` is missing/omitted in the SDK, the `extra.signal` context is mistakenly treated as arguments, breaking the cascade.
- **Validation:** Running `verify_phase84_weather_resilience.ts` MUST capture the string `[TMD:*] ... err=TMD API aborted` in the system logs or stderr.

## 2. No Late Stalls

- **Requirement:** A canceled request must completely drop out of the Node.js event pool within 500ms of the abort signal.
- **Validation:** The `out.log` file generated by the verifier must verify that no "zombie" timers continue firing after the test suite concludes.

## 3. Pro-Wording Compliance

- **Requirement:** Simulated timeouts or upstream failures must NEVER leak debugging strings like `Error fetching from X`. They must be caught and rendered cleanly as `ERR:WX_UPSTREAM_ERROR` in the `structuredContent`.
- **Requirement:** The text field provided to the UI must just say "ขออภัย ยังไม่สามารถดึงข้อมูลอากาศได้ในขณะนี้..."
